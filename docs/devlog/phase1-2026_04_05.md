## What I learned

**C++20 modules and legacy C headers don't mix via `import`.**
SDL2 and standard library headers like `<cstdint>` and `<stdexcept>` cannot be imported as header units with MSVC — macros like `SDL_QUIT` don't survive the translation. The correct pattern is the global module fragment: place all external `#include` directives above the `export module` declaration, between `module;` and the module name. Reserve `import` exclusively for project modules.

```cpp
module;
#include <SDL.h>
#include <cstdint>      // all external headers go here

export module engine.renderer;

import engine.input;    // only project modules go here
```

**Module implementation units must be registered in CMakeLists.txt.**
The `.ixx` interface files go in `FILE_SET CXX_MODULES`. The `.cpp` implementation files go in the plain `PRIVATE` source list. Forgetting the `.cpp` files compiles cleanly but produces linker errors — the declarations exist, the definitions don't.

```cmake
target_sources(pacman
    PRIVATE
        src/main.cpp
        src/engine/renderer.cpp   # implementation units here
        src/engine/game.cpp
    PRIVATE FILE_SET CXX_MODULES FILES
        src/engine/renderer.ixx   # interface units here
        src/engine/input.ixx
        src/engine/game.ixx
)
```

**SDL2 DLL deployment via CMake POST_BUILD.**
The correct way to deploy a runtime DLL is a `POST_BUILD` custom command using CMake generator expressions — not PATH manipulation or debugger working directory settings. This makes DLL deployment part of the build itself, so it works for anyone who clones the repo.

```cmake
add_custom_command(TARGET pacman POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E copy_if_different
        $<TARGET_FILE:SDL2::SDL2>
        $<TARGET_FILE_DIR:pacman>
)
```

**`CMakeSettings.json` is local, not committed.**
Visual Studio generates this file automatically. It belongs in `.gitignore` alongside `.vs/`. If accidentally committed, use `git rm --cached CMakeSettings.json` to stop tracking it without deleting it locally.

**Visual Studio 2026 uses generator string `"Visual Studio 18 2026"`.**
The CMake generator name increments with each major VS release. VS 2022 was version 17, VS 2026 is version 18. Easy to get wrong when copying examples from the web.

```bash
cmake -B build -G "Visual Studio 18 2026" -A x64
```

## What surprised me

The global module fragment is not a workaround — it is the standard-prescribed solution for mixing legacy C headers with C++20 modules. The spec anticipated this problem from the start. Understanding this reframed the toolchain friction from "modules are broken" to "modules are correctly separating concerns."

Also: how quickly the project structure became clear once the foundational documents (`PROJECT.md`, `CODING_STANDARDS.md`) were written first. Having explicit goals and conventions before writing code removed a significant amount of second-guessing during implementation.
