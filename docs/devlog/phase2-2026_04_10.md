## What I learned

**`std::visit` requires a single callable that handles every type in the variant.**
A single lambda has one `operator()` — it cannot be overloaded. The `overloaded` helper solves this by inheriting from multiple lambdas simultaneously and pulling all their `operator()` implementations into a single scope via a pack expansion:

```cpp
template<typename... Ts>
struct overloaded : Ts... { using Ts::operator()...; };
```

The result is one object with multiple `operator()` overloads, which the compiler uses for normal overload resolution. `std::visit` gets its single callable; the correct lambda fires for each type. This pattern is so common it nearly made the standard library — it appears verbatim in the cppreference example for `std::visit`.

**`const WallQuery auto&` is abbreviated template syntax — `auto` is not redundant.**
`WallQuery` is a predicate, not a type. `auto` is still the type placeholder; the concept constrains what it can resolve to. The two forms below are identical:

```cpp
void update(float dt, const WallQuery auto& world);   // abbreviated

template<WallQuery W>
void update(float dt, const W& world);                // explicit
```

At the call site, the caller provides a concrete type. The compiler checks it satisfies `WallQuery` and instantiates the function. Without `auto`, the compiler looks for a type named `WallQuery` — which doesn't exist — and rejects it.

**The abbreviated template syntax breaks down inside `requires` expressions.**
At a function definition, the caller provides the concrete type at instantiation time. Inside a `requires` expression, there is no caller — the compiler would have to invent a type satisfying the concept from thin air, which it cannot do:

```cpp
// Broken — compiler has no concrete type to check against
concept Updatable = requires(T t, float dt, const WallQuery auto& world) {
    { t.update(dt, world) } -> std::same_as<void>;
};
```

The fix is to make the world type an explicit template parameter on the concept itself:

```cpp
concept Updatable<T, W> = WorldQuery<W> && requires(T t, float dt, const W& world) {
    { t.update(dt, world) } -> std::same_as<void>;
};
```

Now the concept has a concrete `W` to substitute. The mental model: `const WallQuery auto&` at a function signature means *"caller provides the type"*. Inside `requires`, there is no caller. The explicit template parameter is what provides it instead.

**Template methods force their definitions into the module interface unit.**
A template is a recipe, not compiled code. The compiler can only instantiate `update<Map>` if the full definition is visible at the call site. A `.cpp` implementation file is compiled in isolation — by the time the linker runs, instantiation is too late. The definition must live in the `.ixx` interface unit, for the same reason `std::vector`'s implementation lives in headers.

This is the general rule for this project: non-template methods → declaration in `.ixx`, definition in `.cpp`. Template methods → definition must be in `.ixx`.

**Concepts should constrain generic boundaries, not encode specific dependencies.**
Attempting to express `Pacman`'s dependency on `Map` through the `Updatable` concept caused cascading problems: `update` became a template, which forced the definition into `.ixx`, which required impl-splitting workarounds. The concept was being asked to do two jobs at once — define the contract *and* express a dependency. These are different concerns.

**Separating collision resolution from position integration produces a cleaner design.**
Rather than `update` receiving the world as a parameter, introducing an explicit `resolveWorldCollisions` step separates two distinct responsibilities:

- `resolveWorldCollisions(const Map& map)` — asks the world what movement is legal, zeroes out illegal velocity components
- `update(float dt)` — integrates position from whatever velocity remains

```cpp
void Pacman::resolveWorldCollisions(const Map& map) {
    if (map.is_wall(x_ + dx_, y_)) dx_ = 0.f;
    if (map.is_wall(x_, y_ + dy_)) dy_ = 0.f;
}

void Pacman::update(float dt) {
    x_ += dx_ * dt;
    y_ += dy_ * dt;
}
```

`Stage::update` drives the sequence explicitly:

```cpp
pacman_entity_.handleInput(input);
pacman_entity_.resolveWorldCollisions(map_);
pacman_entity_.update(dt);
```

`update` is now a plain non-template method. Definition lives in `.cpp`. No linker issues, no workarounds. `Updatable` stays simple.

**`WallQuery`, `PortalQuery`, and `WorldQuery` concepts are not needed yet.**
`resolveWorldCollisions` takes `const Map&` directly. There is no generic parameter to constrain. Introducing a concept here would mean creating a template just to have somewhere to hang the concept — the tail wagging the dog. The concepts would earn their place the moment two or more concrete types needed to be treated interchangeably as world query sources. That moment has not arrived.

## What surprised me

The template linker error was a signal, not just an obstacle. It was the compiler's way of saying that the design was wrong — specifically, that `Updatable` was overreaching by encoding a concrete dependency. Once the concept was corrected, the linker error disappeared along with the design problem.

Also: the realisation that `resolveWorldCollisions` doesn't need to be part of any concept. `Stage` knows about `Pacman` and `Map` concretely — it is not a generic system and is not required to be. Not every method on an entity needs to appear in the concept contract. Concepts define the minimum interface for *generic* systems; `Stage` is not one.

## Blockers / open questions

- `resolveWorldCollisions` is designed but not yet implemented in `.cpp`. Straightforward T3 work for the next implementation session.
- `Pacman::getBounds()` still returns placeholder values. Needs real AABB calculation before collision detection between entities is meaningful.
- The `WallQuery`/`WorldQuery` concepts were removed. They should be reintroduced if and when a second concrete world query source appears — for example, a test double or a procedurally generated map in a later phase.
- Open question for Phase 3: the coroutine-based ghost AI will also need world collision resolution. The design established here (`resolveWorldCollisions(const Map& map)`) should extend naturally — confirm this holds when the first ghost type is introduced.
