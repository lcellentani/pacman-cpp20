## What I learned

**`FetchContent` with `GIT_SHALLOW TRUE` is the correct default for dependencies you don't modify.**
Without it, CMake clones the full commit history of the target repo. For ImGui this caused the generation step to hang. `GIT_SHALLOW TRUE` fetches only the commit at the specified tag — dramatically faster and sufficient for any vendored dependency.

```cpp
FetchContent_Declare(
    imgui
    GIT_REPOSITORY https://github.com/ocornut/imgui.git
    GIT_TAG        v1.91.9b
    GIT_SHALLOW    TRUE
)
```

**ImGui's demo window requires `imgui_demo.cpp` — it is not part of the core library.**
The linker error for `ImGui::ShowDemoWindow` is a common first-time mistake. The demo is deliberately separated so shipping builds can exclude it. Always add it explicitly to the static library target during development.

**ImGui lifecycle has a strict order that mirrors init and teardown.**
Init order: create SDL window → create SDL renderer → `ImGui::CreateContext` → init backends. Teardown order is the exact reverse. Getting this wrong produces subtle crashes or resource leaks that are hard to diagnose. Expressing it inside `Renderer`'s constructor and destructor makes the RAII ownership unambiguous.

**`DebugView::draw` doesn't need a `Renderer&` parameter.**
ImGui maintains its own render context initialised once at startup. `DebugView` populates ImGui draw lists directly — it has no dependency on the SDL renderer. The flush to the screen happens in `Renderer::imgui_render()`, which `Stage` calls explicitly after all debug content is submitted. The boundary is clean: `DebugView` owns the what, `Renderer` owns the when.

**Edge detection for a toggle requires tracking previous frame state.**
`SDL_GetKeyboardState` returns the current held state, not a rising edge. A toggle driven directly from it fires every frame the key is held. The fix is one bool in `Stage`:

```cpp
if (input.debug_toggle && !prev_debug_key_)
    debug_.toggle();
prev_debug_key_ = input.debug_toggle;
```

**`tile_at_index(row, col)` is a cleaner accessor than `tile_at(px, py)` for non-gameplay queries.**
The pixel-coordinate version exists for collision detection — it encodes game logic about tile size. The debug view has no business knowing about pixel coordinates or `TILE_SIZE`. A direct index accessor expresses the right abstraction for a system that just wants to iterate the map grid.

**`PacmanDebugState` belongs in `game.types`, not `engine.types`.**
`engine.types` holds geometric primitives with no game semantics — `AABB` belongs there. `PacmanDebugState` is game-specific vocabulary. Putting it in `engine.types` would leak game knowledge into the engine layer. `game.types` is the correct shared vocabulary module for anything game-layer modules need to exchange.
