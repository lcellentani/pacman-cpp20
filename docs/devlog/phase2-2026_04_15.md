Movement architecture and correctness session. ## What I learned

**Float position models for tile-based movement carry implicit fragility.**
The original implementation tracked position as a `Vec2` float and detected tile centers via integer modulo on a cast float. This works until it doesn't — a fractional drift of even `0.0001f` causes the modulo to land on the wrong value silently. The correct model eliminates the question entirely by making tile-center detection structurally exact.

**The integer tile model: `col_`, `row_`, `offset_`.**
Position is three integers: which tile, and how many pixels into the current inter-tile move. "At tile center" becomes `offset_ == 0` — an exact integer comparison, no tolerance needed. Pixel position for rendering is always derived on demand:

```cpp
int pixel_x() const { return col_ * TILE_SIZE + offset_ * current_dir_.x; }
int pixel_y() const { return row_ * TILE_SIZE + offset_ * current_dir_.y; }
```

The derived nature is important: there is no stored float to drift.

**`offset_` must be gated on having an active direction.**
The first version of the integer model incremented `offset_` unconditionally every tick, even when `current_dir_` was `{0,0}`. This meant `offset_` drifted away from zero while Pac-Man was stationary, so the first move after game start — and after every wall stop — was delayed by up to `TILE_SIZE` frames waiting for `offset_` to lap back to zero. The fix is a single guard:

```cpp
if (current_dir_.x == 0 && current_dir_.y == 0)
    break;
```

This restores the invariant that `offset_ == 0` reliably means "at tile center and ready to decide."

**Opposite-direction reversal requires mirroring `offset_` and advancing the tile coordinate.**
`offset_` encodes displacement *away from `col_/row_` in the direction of `current_dir_`*. When direction flips mid-move, both sides of that contract break simultaneously. The fix repairs both:

```cpp
if (is_opposite(queued_dir_, current_dir_)) {
    col_ += current_dir_.x;
    row_ += current_dir_.y;
    current_dir_ = queued_dir_;
    offset_ = TILE_SIZE - offset_;
}
```

Without the tile advance, `pixel_x/y` jumps by `2 * offset_` in a single frame — the visual pop the bug produced.

**`is_opposite` must guard against the zero direction.**
The original implementation evaluated `is_opposite({0,0}, {1,0})` as true because the clause `(a.y == -b.y && a.x == 0)` matches when both y-components are zero and `a.x` is zero — which is always true for a stationary queued direction paired with horizontal movement. The fix is a single early return:

```cpp
if (a.x == 0 && a.y == 0) return false;
```

No direction is not opposite to anything. Intent is a precondition for the reversal shortcut.

**Passing the map at `reset()` rather than construction.**
The map dependency was moved from the constructor to `Pacman::reset(const Map& map)`. This separates resource acquisition from game state initialisation — `Pacman` can exist without a map, it just cannot play without one. The stored member becomes `const Map*` rather than `const Map&`, since a reference cannot be rebound after construction. The practical consequence is small: all map methods called by `Pacman` are already `const`, so the immutability contract is fully preserved through the pointer. An `assert` in `can_move` enforces the precondition at debug time:

```cpp
bool Pacman::can_move(int col, int row, Dir dir) const {
    assert(map_ && "can_move called before reset()");
    return !map_->is_wall_at(col + dir.x, row + dir.y);
}
```

The benefit scales directly to Phase 3: four ghost constructors that take no arguments, map dependency threaded once in `Stage::reset()`.

## What surprised me

The reversal jump bug was a direct consequence of the model's internal contract, not a logic error. `offset_` has a directional assumption baked into it that holds perfectly as long as direction only changes at `offset_ == 0`. The opposite-direction shortcut is the one case that breaks the contract mid-move — and the fix had to repair both sides explicitly. Once the contract was stated clearly, the fix was obvious.

The `is_opposite` false positive was subtle: it only triggered on horizontal movement paired with a zeroed queued direction, and only because `0 == -0` is true. It would have been invisible in normal play until the exact moment Pac-Man stops against a horizontal wall and the player hasn't pressed anything yet.
