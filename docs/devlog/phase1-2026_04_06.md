## What I learned

**A concept is not a type — it cannot be used as a parameter type directly.**
The instinct to write `void onCollision(Collidable& other)` is wrong. Concepts are compile-time predicates; they have no runtime representation. The correct pattern is a concept-constrained template:

```cpp
void onCollision(Collidable auto& other);
```

**`std::visit` accepts multiple variants — this is how you get double dispatch.**
Passing two variants to `std::visit` dispatches on the active type of both simultaneously. This is the clean, modern solution to the classic double dispatch problem, with no virtual functions and no type tags:

```cpp
std::visit(overloaded{
    [](Pacman& p, Ghost& g)  { resolveCollision(p, g); },
    [](Ghost& g,  Pacman& p) { resolveCollision(p, g); },
    [](auto&, auto&) { }
}, gameObjects[i], gameObjects[j]);
```

Both orderings must be covered because the active type combination depends on insertion order, not on game logic.

**`static_assert` is the bridge between types and concepts.**
Without it, concept violations surface deep inside template instantiations — hard to read, hard to locate. `static_assert` forces the check to the type definition boundary, where the error message is clean and the fix is obvious:

```cpp
static_assert(GameEntity<Pacman>, "Pacman must satisfy GameEntity");
```

**Concepts compose — and composition is where they become design vocabulary.**
Three fine-grained concepts (`Drawable`, `Updatable`, `Collidable`) can be composed into a single domain-level concept (`GameEntity`). This is more than syntactic convenience — it makes architectural intent legible:

```cpp
template<typename T>
concept GameEntity = Drawable<T> && Updatable<T> && Collidable<T>;
```

**The variant itself can be guarded at compile time.**
A small template struct over the variant's type list enforces that every type in `GameObject` satisfies `GameEntity`. Adding a non-conforming type is a compile error with a readable message:

```cpp
template<typename... Ts>
struct AllGameEntities<std::variant<Ts...>> {
    static constexpr bool value = (GameEntity<Ts> && ...);
};

static_assert(AllGameEntities<GameObject>::value,
    "Every type in GameObject must satisfy GameEntity");
```

## What surprised me

Concepts in the current sketch are mostly doing documentation and light enforcement work — the real architectural weight is carried by `std::variant` and `std::visit`. This reframed the question from "how do I use concepts?" to "where do concepts actually add value that the variant doesn't already provide?" The answer: at the type definition boundary, via `static_assert`, and as composed domain vocabulary. That's a narrower but more honest role than initially assumed.

Also: the `component_marker` tagging approach — using a nested type alias to filter variants by semantic role — was an understandable instinct but the wrong tool. The variant already knows the concrete type; `std::visit` already dispatches on it. The tag was reinventing information that was already available for free.
